# Exploration Log — Adult (Census Income) Dataset

This is a running narrative of the exploratory work as it happened: what was tried, what it showed, and what that led to next. It is written as the analysis happens, not reconstructed afterward — dead ends are kept in because they shaped what got tried next, not because they made the final report.

## Setup

Downloaded `adult.data` (the documented 32,561-row training split) and `adult.names` (the codebook) directly from the UCI repository. Also pulled `adult.test` (16,281 rows) out of curiosity — the two splits together form the full 48,842-row "adult.all" the codebook describes — but the task was scoped to `adult.data`, so `adult.test` was set aside and is not used in this analysis. Noting it exists in case a train/test consistency check is ever wanted later.

R was available locally (4.6.1) with `tidyverse`, `arrow`, and friends already present in the user library; `renv::hydrate()` linked what the project needed into an isolated project library in seconds rather than reinstalling from source. Quarto wasn't on PATH as a standalone CLI, but RStudio ships its own bundled copy (`/Applications/RStudio.app/Contents/Resources/app/quarto/bin/quarto`) — used that instead of installing a second copy.

## Phase 1 — Data Quality Checks

Ran a first-pass script (`R/01_data_quality.R`) reading the raw CSV with explicit column names from the codebook (the file itself ships headerless) and `na = character()` so nothing gets silently coerced to NA on read — the point of this pass is to *see* what's actually in each cell before deciding what counts as missing.

**Row count matched exactly**: 32,561 actual vs. 32,561 documented. No surprises there.

**The first real finding: missingness is disguised, not absent.** A raw `is.na()` check across all 15 columns came back completely clean — zero NAs anywhere. That would normally read as "no missing data," but the codebook explicitly says Unknown values were encoded as the literal string `"?"` before this file was published. Re-checked for that literal specifically and found it concentrated in exactly three columns: `workclass` (1,836), `occupation` (1,843), and `native_country` (583). If the first check had stopped at `is.na()`, this would have been reported as complete data — it isn't. Handling decision: recode `"?"` to real `NA` in the cleaned artifact (done in the same script, written to `data/processed/adult_clean.parquet`), so downstream numeric/categorical summaries don't silently treat "?" as its own factor level.

**`workclass` and `occupation` missingness looked like it might be the same rows** — both around 1,840 missing, both plausibly describing "not currently employed." Worth checking directly in Phase 2 rather than assuming from the similar counts.

**Numeric ranges all came back plausible** — `age` 17–90, `hours_per_week` 1–99, `education_num` 1–16 — no impossible values (no negative ages, no zero-length work weeks, nothing outside a defensible range for a CPS extract that the codebook says was already filtered to `age > 16`).

**`capital_gain` has an unmistakable ceiling.** 159 of 32,561 rows sit at exactly 99,999 — not "high," but bit-for-bit identical to the maximum. That pattern (a spike of exact duplicates at the top of a continuous range) is the signature of income top-coding — a very standard Census Bureau disclosure-avoidance practice where amounts above a threshold get truncated to a single ceiling value rather than reported exactly, to keep any one very-high-income respondent from being re-identifiable. Flagging this now as something to characterize properly in Phase 2's outlier check, and to name explicitly as a downstream-modeling consideration (a model treating 99,999 as a real quantity rather than "topcoded, true value unknown but at least this much" would be misreading it).

**`capital_loss`'s max (4,356) does not show the same pattern** — only 3 rows sit at that value, not a large spike, so it doesn't look like a comparable topcode; treating it as a genuine (if rare) high observed value rather than censored, pending the Phase 2 distribution check.

**`fnlwgt` is not a personal attribute — it's a sampling weight**, and its range (12,285 to 1,484,705) reflects that: it's the Census Bureau's estimate of how many people in the actual U.S. population this one CPS respondent's record represents, not a fact about the respondent. This gets flagged explicitly so it doesn't get treated as an ordinary numeric feature anywhere downstream — averaging or correlating it against age/hours/income directly would conflate a sampling artifact with a demographic fact.

**`education` and `education_num` are perfectly redundant** — every one of the 16 education strings maps to exactly one numeric level, in ascending order (Preschool=1 … Doctorate=16). Confirmed by checking `distinct(education, education_num)` came back with exactly 16 rows, one per level, no crossover. Useful to know before Phase 2/3: any ordered-education analysis can use `education_num` directly instead of re-deriving an ordinal encoding from the string column.

**24 exact duplicate rows** (every column identical, `income` included) — small relative to 32,561, but the codebook's own summary line ("Duplicate or conflicting instances: 6") describes a smaller number for the *combined* train+test file, so 24 duplicates found on the train file alone doesn't cleanly reconcile with that documented figure. Not resolving that discrepancy by guessing — naming it as an open question rather than assuming either number is wrong.

**Class balance**: 24,720 `<=50K` vs. 7,841 `>50K`, i.e. 24.08% positive. The codebook states 23.93% (with unknowns) / 24.78% (without) for the *combined* train+test file — the train-only figure computed here (24.08%) sits right between those two, which is what's expected if the train split is a representative random subsample of the combined file. Consistent, not a discrepancy.

## Phase 2 — General EDA

**Missingness pattern, resolved.** The "is `workclass`-NA the same 1,836ish rows as `occupation`-NA" question from Phase 1 came back cleanly: every single `workclass`-NA row is also `occupation`-NA. The extra 7 rows where only `occupation` is NA turned out to be exactly the 7 people with `workclass == "Never-worked"` — which makes complete sense (no job, no occupation to report, but the fact of never having worked is itself a known "employer type"). So `workclass`/`occupation` missingness is structural, not two independent random processes. `native_country`'s missingness (583 rows) barely overlaps with the other two (only 27 rows in common) — that one really does look scattered/independent, as the "report even if null" line for this check would put it.

**Tried assuming `workclass`-NA meant "not currently working," and that assumption didn't survive contact with the data.** The natural read of "employer type unknown" is "not in the labor force" — but this dataset's own extraction criteria (stated in the codebook) already required `hours-per-week > 0` for every row, so a genuinely non-working respondent couldn't appear at all. Checked directly: mean `hours_per_week` among `workclass`-NA rows is 31.9, and *zero* of them report 0 hours. So these are people working real hours who didn't (or couldn't) name what kind of employer they have — a different, more specific condition than "unemployed." Their 10.4% >$50K rate sits much closer to the near-zero rates of `Never-worked`/`Without-pay` than to `Private`'s 21.9%, which is suggestive but not fully explained yet — flagged as a Phase 3 candidate rather than resolved here.

**`hours_per_week`'s IQR-based outlier count (27.7%) looked alarmingly high at first** — almost a third of the column flagged as "outliers" would normally be a red flag. Checked the shape directly rather than trusting the IQR rule at face value: the variable has a very strong central mode at exactly 40 with a long but entirely plausible two-sided tail (1 hour up to 99 hours). The IQR method is penalizing a tight, spiky center, not actually finding bad data — decided this needs to be shown as a plain distribution (so the spike-plus-tail shape is visible), not framed as an outlier problem, when it reaches the report.

**`capital_gain`'s topcoding spike (159 rows at exactly 99,999, first spotted in Phase 1) checks out as real censoring, not noise.** All 159 of those topcoded rows carry the `>50K` label — a coherent, sensible pattern for Census-style disclosure-avoidance censoring (top incomes get capped rather than published exactly), not something that looks like a data artifact unrelated to the outcome.

**Correlation among the numeric columns came back weak across the board** (max |r| = 0.148, `education_num`–`hours_per_week`) — no near-redundant numeric pairs to worry about.

**Categorical-association check turned up a size ordering worth noting**: `marital_status`'s association with `income` (Cramér's V = 0.447) is nearly twice as strong as `sex`'s (0.216), and `sex`'s association with `occupation` (0.434) is almost as strong as `marital_status`'s with income. That ordering — marital status and occupational segregation looking like the stronger stories, sex-on-its-own looking comparatively weaker — became the seed for the next line of digging.

**Followed the marital-status lead and found the most counter-intuitive result of the whole analysis.** Split `Married-civ-spouse` by sex: 44.6% of married men clear $50K vs. 45.5% of married women — essentially the same rate, despite wives working noticeably fewer average hours (36.7 vs. 44.1) and despite the aggregate sex gap being enormous (30.6% men vs. 11.0% women, dataset-wide). Went one level further and looked at the `relationship` column's `Wife` category specifically (n=1,566): 47.5% >$50K, fractionally *above* `Husband`'s 44.9%. First reaction was to suspect a bug in the grouping code — re-ran the aggregate `sex`-only tabulation side by side with the `Married-civ-spouse`-only tabulation to make sure the same underlying data produced both numbers, and it does. The resolution: the aggregate female rate is dragged down by the much larger population of women who are `Own-child` (1.1% >50K, mean age 24.6 — mostly young adults still living at home) and `Unmarried`/`Not-in-family` (4.2%/7.3%) — not by an income penalty *within* marriage. The sex gap is real in aggregate, but it is concentrated almost entirely outside the married-couple subpopulation, not spread evenly across all women.

## Phase 3 — Domain-Driven Questions

Ran the structural sweep (every categorical variable against `income`, every anomaly already flagged, every planned Phase 0 benchmark, the sex/marital-status interaction just found) methodically rather than free-associating, and logged the full candidate pool to `R/candidate_questions.md` before choosing what goes in the report — so the fuller set is visible on disk regardless of which 4-6 make the final cut. Selected: education threshold-crossing, occupation ranking (pending an authoritative reclassification once Phase 0's occupation-scheme research returns), the sex/marriage interaction just described, the `workclass`-missing population's real condition, `capital_gain` topcoding as a modeling flag, and the $50K-threshold-in-1994-context external benchmark.

## Phase 4 — Charts

Built the six charts backing the selected questions (`R/05_charts.R`), each cached to `data/processed/` as the data feeding it (Parquet) plus the finished ggplot object (`.rds`), per the caching discipline — the report loads these back rather than recomputing or re-plotting from raw data on every render. Sanity-checked each one by rendering to a throwaway PNG and viewing it directly before wiring it into the report, rather than trusting the object saved without error meant it would actually display correctly.

One design decision worth recording: the sex/marriage finding doesn't fit any of the four named chart patterns exactly (it's not skewed counts needing a log axis, not a ranking, not an ordinal scale reference). Adapted the dumbbell idea without the log-axis part — two points per category connected by a segment, comparing "All respondents" against "Married, spouse present" — since the five-second question ("does the gap look different in these two scopes?") is answered by point position and connecting-segment length, not by any log transform. This is exactly the kind of case `visual-style.md` describes as adapting a familiar element rather than inventing a new one.

## Phase 0 — External Grounding (returned)

Dispatched as a background research task at the start of Phase 1/2 (per the skill's parallelization guidance — Phase 0 and Phase 1 don't depend on each other) and returned once Phase 2-4 work was already well underway. Findings, all from primary Census Bureau tables (parsed directly, not paraphrased from a secondary source) unless flagged otherwise:

- **Household vs. individual income, 1994**: median household income $32,264 (Census P60-189); $50K sat inside the 4th household-income quintile (boundary $62,841), roughly the top 30% of households. But this dataset's target is *individual* income, and the individual-level benchmark tells a different story: median annual earnings for full-time, year-round *male* workers were only $30,854 in 1994 (Table P-38) — the higher-earning sex by this measure didn't clear $31K. That reframes the whole analysis: $50K individual income in 1994 was a demanding, not a modest, bar — well above what a typical full-time worker of either sex earned. This directly explains why this dataset's 24% positive rate isn't itself surprising once the extraction filters (working adults only, `hours-per-week > 0`) are accounted for.
- **Gender earnings ratio, 1994**: Census Table P-38 gives women's median full-time annual earnings at 72.0% of men's ($22,205 / $30,854). The research agent also surfaced a commonly-cited BLS weekly-earnings ratio (76.4%) but flagged it explicitly as **not independently verified** this session (every bls.gov fetch attempt returned HTTP 403) — decided not to cite that figure in the report since it didn't clear the same "read the primary document myself" bar as everything else, even though it's probably fine. Used the verified Census P-38 figure only.
- **Black-white income gap, 1994**: Census Table H-5 gives a household median income ratio of 61.8% (Black $21,027 / White $34,028) — household-level, used in the report only to confirm the *direction* of this dataset's own race gap, not its magnitude (different statistic, different income unit).
- **Occupation classification — the most useful surprise of the whole Phase 0 pass.** Went in expecting to need an external scheme to *collapse* the dataset's 14 raw occupation levels (that was the plan going into Phase 3). Instead, Census Technical Paper 65 shows the dataset's 14 labels are a near-verbatim match to the Census Bureau's own 1990 scheme of 13 major occupation groups (plus Armed Forces, kept separate). So the actual finding is the opposite of what was expected — nothing needs collapsing, because the raw coding already *is* the authoritative external classification. Revised the "Occupation, Reclassified" report section (planned as a collapsing exercise) into "Occupation, Already Authoritatively Classified" to reflect what was actually found, rather than forcing the original plan onto a result that contradicted it.
- **`fnlwgt` methodology**: independently confirmed via the CPS's own technical documentation (not this dataset's docs) — multi-stage weighting (base weight, noninterview adjustment, raking to state/Hispanic-origin/race-age-sex population controls). Consistent with, and independent of, what `adult.names` itself already said — used as corroboration in the data dictionary note rather than as a new finding.
